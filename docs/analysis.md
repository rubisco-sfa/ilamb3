# Analyses

Writing an analysis module in `ilamb3` can be complex because it needs to be able to handle a wide range of datasets that users may provide. We tend to develop analysis ideas in a standalone script with a few specific data sources in mind and may be unaware of all the variances that are common across CMIP models and/or reference data sources. In this tutorial we will explore a simple analysis idea and start implementing it simplistically, but then gradually accommodate dataset differences to make the analysis more general.

## Getting Started

For simplicity, imagine you wanted to write an analysis that returns the absolute relative error between a variable in the reference and comparison datasets,

$$\varepsilon = \frac{|v_{\mathrm{ref}} - v_{\mathrm{com}}|}{|v_{\mathrm{ref}}|}.$$

There are many ways that you could implement this idea. However, for integration with `ilamb3`, we ask that you consider the following function prototype as a target for your analysis function.

```python
def my_analysis_func(ref: xr.Dataset, com: xr.Dataset) -> tuple[pd.DataFrame, xr.Dataset, xr.Dataset]:
```

This means that your analysis function should:

- Take in two `xr.Dataset` objects as inputs, one as a reference and another as a comparison.
- Return a `pd.DataFrame` that will contain any scalars or scores generated by the analysis. While this dataframe can have as many rows/scalars as you like, the `ilamb3` system will expect specific columns which we will explain below.
- Also return two `xr.Dataset` objects that will contain any intermediate datasets with information that you wish to plot. If your analysis does not have such datasets, you can return empty `xr.Dataset` objects.

This may seem like overkill for our simple target example, but this abstraction allows for flexibility in implementing more complex ideas in the same framework. The following particularizes the function prototype for our sample application.

```{code-cell} python
import numpy as np
import pandas as pd
import xarray as xr

def absrelerr(ref: xr.Dataset, com: xr.Dataset, varname: str) -> tuple[pd.DataFrame, xr.Dataset, xr.Dataset]:

    # naive mean absolute relative error
    eps = (np.abs(ref[varname] - com[varname]) / np.abs(ref[varname])).mean()

    # build up the output dataframe
    df = pd.DataFrame(
        [
            {
                "source": "Comparison",
                "region": "None",
                "analysis": "Relative Error",
                "name": "Mean Absolute Relative Error",
                "type": "scalar",
                "units": "1",
                "value": float(eps), # to change type from xr.DataArray
            }
        ]
    )

    return df, xr.Dataset(), xr.Dataset()
```

Note that in order for our function to work, we included another argument `varname` which is the name of the variable that is expected to be in both input datasets. As you develop your own analysis function, you may find a need to include this and other keywords to keep the function general. This is fine and we will later address what to do with all such keyword arguments. This code also illustrates that the returned dataframe should contain the following columns:

- `source`: The name of the source to which this scalar belongs, either `Reference` or `Comparison`. In general you may wish to report values that involve each source individually. Scalars which use both reference and comparison datasets are assigned to the `Comparison` source.
- `region`: The region label over which the scalar applies. In general `ilamb3` analyses can be performed over multiple regions. For now we will leave this simply as `None` to indicate no region is specified.
- `analysis`: The name of the analysis. In the `ilamb3` output, there will be sections for each analysis performed and this label will be used to identify which scalars correspond to which analysis. This should be consistent for all scalars in the same analysis.
- `name`: The name of the scalar. This should be a descriptive name for the scalar, such as `Mean Absolute Relative Error` as it will appear in the table of scalars in the `ilamb3` dataset pages.
- `type`: The type of scalar being reported, either `scalar` or `score`. A `score` in the `ilamb3` parlance is a synthesis of model performance on the unit interval and will be included in an overall composite score.
- `units`: The units of the scalar, dimensionless.
- `value`: The value of the scalar, `eps`.

In order to test we create two small datasets using a function from our testing suite. You may have local data available to use instead.

```{code-cell} python
from ilamb3.tests.test_run import generate_test_dset

ref = generate_test_dset(nyear=2, nlat=10, nlon=20, name="tas", unit="degC")
com = generate_test_dset(nyear=2, nlat=10, nlon=20, name="tas", unit="degC", seed=2)

df, _, _ = absrelerr(ref, com, "tas")
df
```

We see that our analysis returns a dataframe with a single row containing the relative error between the reference and comparison datasets. The value may seem high, but our test data is coarse and random and so at the moment we take this as a sign that the implementation is working as expected.

## What Could Go Wrong

You may have already noticed some potential problems with our implementation. By subtracting `com[varname]` from `ref[varname]`, we are assuming that these two dataarrays are indexed the same way. In practice, this is not usually true and the cause of a lot of headache. Here are a few differences we commonly see:

1. Both datasets may contain a `time` dimension but use different calendars which are incompatible.
2. The time ranges could be different. For example, reporting a relative error from a reference dataset that spans the 1980's is not meaningful if the comparison dataset was only over the 1850's.
3. The datasets may be in different, but compatible, units.
4. The datasets may both be gridded but call their latitude and longitude dimensions different names (i.e. `lon` versus `longitude` or `Longitude`).
5. While the two datasets may both be global, they will seldom be on the same grids (i.e. values of latitude and longitude). Even if they are off by a tiny margin, `xarray` will treat them as different and yield results which are broadcast `nan` to the missing dimensions.

While you are ultimately responsible for handing these and other possible variances in input datasets,`ilamb3` implements some functions that encode logic that we use in our comparisons to handle these differences. These are found in two modules in the `ilamb3` package:

- `ilamb3.dataset` contain functions which operate on a single dataset.
- `ilamb3.compare` contain functions which embed logic for comparing two or more datasets.

To use these functions, import them from the `ilamb3` package:

```python
import ilamb3.compare as cmp
import ilamb3.dataset as dset
```

Let's use these functions to expand this line above:

```python
# naive mean absolute relative error
eps = (np.abs(ref[varname] - com[varname]) / np.abs(ref[varname])).mean()
```

into the following lines which will now handle many of the potential issues and shortcomings of our first implementation.

```python
# trim away non-overlapping time and ensure units/lon are consistent
ref, com = cmp.make_comparable(ref, com, varname)

# remove possible problems with calendars by using integrated time means
if dset.is_temporal(ref[varname]):
    ref[varname] = dset.integrate_time(ref, varname, mean=True)
if dset.is_temporal(com[varname]):
    com[varname] = dset.integrate_time(com, varname, mean=True)

# nest the spatial grids as explained in Collier, et al., JAMES, 2018
ref, com = cmp.nest_spatial_grids(ref, com)

# compute an area-weighted mean
epsmap = np.abs(ref[varname] - com[varname]) / np.abs(ref[varname])
eps = dset.integrate_space(epsmap, varname, mean=True)
```

- `make_comparable` is a catch-all comparison function which we use in many of our analysis routines. It will trim away the portions of time that are non-overlapping and ensure that the units are correct and that longitude is on the same interval (either [-180, 180] or [0, 360]) among other things.
- We integrate away the time dimension by taking a mean using `integrate_time`. This is a design choice, but also circumvents the need to handle calendar differences. By taking this mean I am saying that I am less interested in the temporal differences which should be communicated in the documentation for your new analysis routine.
- We nest the spatial grids using `nest_spatial_grids` (see [*Collier, et al. 2018*](https://doi.org/10.1029/2018MS001354) for details and discussion) to handle the resolution differences. You could also opt to interpolate the grids to a common resolution.
- Finally, we compute the absolute relative error but first leave it as a spatial map called `epsmap`. This we can return later as a quantity to plot. Then the correct mean to take in this case is the integrated (area-weighted) mean of `epsmap`.

From these additions, you can see the complexity is increasing. However, as you are the author of an analysis function, `ilamb3` will not make these decisions for you. It is an important part of your analysis routine, and many times involve assumptions that you have made (consciously or not) and therefore should be made explicitly. Our additions have introduced one more problem: we are integrating over space but the datasets passed in could be collections of sites (such as the Fluxnet collection). While we could handle that case in the logic of the function, let's rather use this as a moment to illustrate another concept. Your analysis routine should detect when your method is not appropriate for the input data and raise an error (`ilamb3.exceptions.AnalysisNotAppropriate`). So for now, we will add the following lines at the top of the function to error if site data is being used.

```python
if dset.is_site(ref[varname]) or dset.is_site(com[varname]):
    raise AnalysisNotAppropriate()
```

At this point we have made several additions to the sample function without showing the full code. Expand the cell below to see all the changes we have made thus far. If you have made it to this point, you are through the hard part of writing an analysis function.

```{code-cell} python
:tag: [hide-cell]
import numpy as np
import pandas as pd
import xarray as xr

import ilamb3.compare as cmp
import ilamb3.dataset as dset
from ilamb3.exceptions import AnalysisNotAppropriate

def absrelerr(
    ref: xr.Dataset, com: xr.Dataset, varname: str
) -> tuple[pd.DataFrame, xr.Dataset, xr.Dataset]:

    # limit analysis to gridded data
    if dset.is_site(ref[varname]) or dset.is_site(com[varname]):
        raise AnalysisNotAppropriate()

    # trim away non-overlapping time and ensure units/lon are consistent
    ref, com = cmp.make_comparable(ref, com, varname)

    # remove possible problems with calendars by using integrated time means
    if dset.is_temporal(ref[varname]):
        ref[varname] = dset.integrate_time(ref, varname, mean=True)
    if dset.is_temporal(com[varname]):
        com[varname] = dset.integrate_time(com, varname, mean=True)

    # nest the spatial grids as explained in Collier, et al., JAMES, 2018
    ref, com = cmp.nest_spatial_grids(ref, com)

    # compute an area-weighted mean
    eps_map = np.abs(ref[varname] - com[varname]) / np.abs(ref[varname])
    eps = dset.integrate_space(eps_map, varname, mean=True)

    # build up the output dataframe
    df = pd.DataFrame(
        [
            {
                "source": "Comparison",
                "region": "None",
                "analysis": "Relative Error",
                "name": "Mean Absolute Relative Error",
                "type": "scalar",
                "units": "1",
                "value": float(eps),
            }
        ]
    )

    return df, xr.Dataset(), xr.Dataset()
```

## Integrating into `ilamb3`

At its core, an `ilamb3` analysis is just a function that operates on two datasets and returns scalars and things to plot. There are a few system needs however that require analyses be slightly more complicated.

1. In order to determine what can be executed from the model output you give, `ilamb3` needs to know beforehand what data will be used. Our example analysis would need to communicate that `varname` is needed in order to execute.
2. In order to call all analyses consistently, we need the functions to always have the same signature. But this does not leave any room for extra parameters that may be important. In the above example, we have `varname` that was added to the signature that needs to be moved somewhere else.

To this end, analyses are not merely functions but an [abstract base class](https://docs.python.org/3/library/abc.html) (ABC) we call an [ILAMBAnalysis](https://github.com/rubisco-sfa/ilamb3/blob/b32815578969cc8d4c7f6a6f58b0beed3b2c6574/ilamb3/analysis/base.py#L16). Lets change the function we wrote above into an `ILAMBAnalysis` and see how they work. If you are unfamiliar with classes in python, you may want to look over an introductory [tutorial](https://realpython.com/python-classes/).

```{code-cell} python
:linenos:
:emphasize-lines: 7,12,13,14,16-21
import numpy as np
import pandas as pd
import xarray as xr

import ilamb3.compare as cmp
import ilamb3.dataset as dset
from ilamb3.analysis.base import ILAMBAnalysis  # import the ABC from ilamb
from ilamb3.exceptions import AnalysisNotAppropriate
from ilamb3.tests.test_run import generate_test_dset


class absrelerror(ILAMBAnalysis):  # define a class that inherits from the ABC
    def __init__(self, varname: str):
        self.varname = varname

    def __call__(
        self,
        ref: xr.Dataset,
        com: xr.Dataset,
    ) -> tuple[pd.DataFrame, xr.Dataset, xr.Dataset]:
        varname = self.varname
        if dset.is_site(ref[varname]) or dset.is_site(com[varname]):
            raise AnalysisNotAppropriate()

        # trim away non-overlapping time and ensure units/lon are consistent
        ref, com = cmp.make_comparable(ref, com, varname)

        # remove possible problems with calendars by using integrated time means
        if dset.is_temporal(ref[varname]):
            ref[varname] = dset.integrate_time(ref, varname, mean=True)
        if dset.is_temporal(com[varname]):
            com[varname] = dset.integrate_time(com, varname, mean=True)

        # nest the spatial grids as explained in Collier, et al., JAMES, 2018
        ref, com = cmp.nest_spatial_grids(ref, com)

        # compute an area-weighted mean
        eps_map = np.abs(ref[varname] - com[varname]) / np.abs(ref[varname])
        eps = dset.integrate_space(eps_map, varname, mean=True)

        # build up the output dataframe
        df = pd.DataFrame(
            [
                {
                    "source": "Comparison",
                    "region": "None",
                    "analysis": "Relative Error",
                    "name": "Mean Absolute Relative Error",
                    "type": "scalar",
                    "units": "1",
                    "value": float(eps),
                }
            ]
        )

        return df, xr.Dataset(), xr.Dataset()
```

Lines that were changed have been highlighted for emphasis. In words, the above code snippet makes the following changes:

- We have imported the abstract base class `ILAMBAnalysis` and made `absrelerror` a class that *inherits* from it.
- We created a member function named `__init__` and moved our `varname` function argument to it. Python has a number of *double under* or *dunder* methods that have special behavior. The `__init__` method is called when an instance of the class is created, and it allows us to initialize the instance's attributes.
- Then we created a member function named `__call__` and put the content from our function inside. The `__call__` method will allow us to call an instance of the `absrelerror` class as if it were a function.

At the moment, it does not yet seem like we have accomplished much by this change. However, if you try to create an instance of this class we have built,

```python
my_analysis = absrelerror("tas")
```
you should see that python throws an error:
```
TypeError: Can't instantiate abstract class absrelerror without an implementation for abstract methods 'plots', 'required_variables'
```

The `ILAMBAnalysis` base class defines a template for all analysis functions. If you don't implement all of the methods we need, you can't even create an instance. You will see that the error message is complaining about not having an implementation for two functions. The second of these is `required_variables`, a function that returns a list of variables that the analysis needs in the input `Dataset`. So we add another member function that returns `varname` in a list. `ilamb3` will use this internally to query model data and prepare the input datasets for the analysis.

```python
    def required_variables(self):
        return [self.varname]
```

The other of these functions is `plots`, a function that generates plots for the analysis. For the moment, add the following implementation that returns an empty dataframe and we will return to this later.

```python
    def plots(
        self, df: pd.DataFrame, ref: xr.Dataset, com: dict[str, xr.Dataset], path: Path
    ) -> pd.DataFrame:
        return pd.DataFrame()
```

At this point your analysis function is ready to be integrated into `ilamb3`.

## Making `ilamb3` aware of your analysis function

When adding a analysis function, you can work in your own directory until you are ready to register it with the `ilamb3` system. To do this you will first need to clone the repository:

```bash
git clone git@github.com:rubisco-sfa/ilamb3.git
cd ilamb3
```

and then move the python file with your analysis function in it into the `ilamb3/analysis` directory:

```bash
mv my_function_file.py ilamb3/analysis/
```

Then you need to edit the `__init__.py` file found in this directory to make sure the system will know about your new analysis function. This should look something like this:

```python
from ilamb3.analysis.my_function_file import absrelerror  # noqa
```

Once you have imported your analysis function into the `__init__.py` file, the `ilamb3` system will register it in its list of analyses.
